# 01 · think：确定性短链的方案

> skill：`think` —— 动手前一次一问拷问边界，产出决策完整的计划。
> 触发：「/think 那就做这个 CLI，给我一个方案」

think 一次只问一个会改变设计的问题，问到没有悬而未决的判断为止。下面先是拷问记录，再是定稿计划。

## 拷问记录（一次一问）

**Q1：短码从哪来？自增计数器，还是 URL 的函数？**
→ 必须是 URL 的纯函数。office-hours 已经把"确定性、可跨机器复算"定成命根子，自增计数器需要中心状态，直接出局。

**Q2：用什么散列？**
→ 加密散列（sha256）。它是纯函数，对同一输入跨进程跨机器恒定。**绝不能用语言内置的、带进程级随机种子的散列**，那种东西进程内稳定、跨进程就变。（记进 decision-log。）

**Q3：短码多长？碰撞怎么办？**
→ 取散列前若干字节 mod 62^6，得 6 位 base62。62^6 ≈ 568 亿，自用量级碰撞概率可忽略。碰撞处理（探测、加长）是镀金，交给 scope-guard 砍。

**Q4：expand 怎么实现？也复算吗？**
→ 散列不可逆，expand 必须查存储。所以 shorten 时把 `code -> url` 写进一个本地 JSON。存储只服务 expand，不参与短码生成 —— 短码生成永远是纯函数。

**Q5：存储放哪？多端怎么同步？**
→ 本地 JSON，路径可用环境变量 `URLSHORT_DB` 覆盖。多端同步不在范围（自用，单机）。

## 定稿计划（决策完整）

**核心前提（不可让步）**：短码 = `base62(sha256(url) 取前 8 字节 mod 62^6)`，纯函数，不掺任何运行时状态。

**模块**：
- `shorten.ts`：纯函数核心（`base62` / `urlHash` / `codeFor`）+ 存储读写 + `shorten` / `expand`。
- `cli.ts`：argv 解析，调核心。
- 存储：`URLSHORT_DB` 指向的 JSON，缺省 app 目录下 `links.json`。

**命令**：`shorten <url>` / `expand <code>` / `list`。

**验证策略**：纯函数层 + 往返层用 `node:test`。**注意**：要专门钉一条"跨进程同码"的测试，因为 Q2 的风险恰恰只在跨进程暴露。（这条当时没写进 tdd，成了 hunt 的伏笔。）

**明确不做**：见 [scope-guard](03-scope-guard.md)。
