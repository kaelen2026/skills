# 05 · tdd：红绿重构把工具写出来

> skill：`tdd` —— 给没人 review 的代码织一张随时能安全重构的网。
> 触发：「/tdd 开始写，边写边测」

tdd 以可观测行为为单位，红 -> 绿 -> 重构，纵向切。下面是这次实际的红绿节奏，以及一个**当时没意识到的盲区**。

## 红绿节奏（真实顺序）

1. **base62 定长** —— 红（函数不存在）-> 写最小实现 -> 绿。钉住"任何输入都返回 6 位、只用字母表字符"。
2. **shorten 返回定长短码** -> 绿。
3. **shorten -> expand 往返还原** -> 绿。钉住存储读写。
4. **同进程内幂等**（同一 URL 两次 shorten 同码）-> 绿。
5. **expand 未知短码返回 undefined** -> 绿。

跑一把，6 条全绿：

```
ℹ tests 6
ℹ pass 6
ℹ fail 0
```

工具上线，自用了几天。

## 盲区：绿不等于对

第 4 条"幂等"测试**只在单个进程内**验证。而第一版 `urlHash` 在模块加载时塞了个随机 salt：

```ts
const SESSION_SALT = randomBytes(8).toString("hex");
// ...
createHash("sha256").update(SESSION_SALT + url)
```

同一个进程里 `SESSION_SALT` 是常量，所以幂等测试**理所当然地绿**。测试覆盖了"幂等"这个词，却没覆盖"**跨进程**幂等"这个真实使用场景。

> tdd 的网只能网住你想到要钉的行为。think 计划里其实写了"要钉一条跨进程同码的测试"，但 tdd 这一步漏了 —— 这正是 hunt 存在的理由：**测试绿了，行为仍可能错**。
