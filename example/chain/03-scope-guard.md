# 03 · scope-guard：这次不做什么

> skill：`scope-guard` —— 决定这次**不做**什么，砍掉镀金，切出 MVP 边界。
> 触发：「/scope-guard 这次别做大」

scope-guard 的产出是一张"不做清单"，每条都写明为什么现在不做、以及什么信号出现时再考虑。

## 不做清单

| 砍掉的东西 | 为什么现在不做 | 什么时候回来做 |
|-----------|--------------|--------------|
| **碰撞处理**（探测/重散列/加长） | 62^6 ≈ 568 亿，自用量级碰撞概率可忽略。处理碰撞要引入状态，和"纯函数确定性"打架。 | 存储量级到百万条、或 list 里真出现一次碰撞。 |
| **自定义短码 / 别名** | 一旦允许人指定短码，"短码是 URL 的函数"这条前提就破了。 | 永远不做，除非工具定位从"确定性"改成"可读别名"。那是另一个产品。 |
| **删除 / 改写命令** | expand 依赖存储只增不改，少一类状态变更就少一类 bug。 | 存储文件大到需要清理时，加 `gc` 而不是 `delete`。 |
| **Web 服务 / 跳转 / 统计** | office-hours 已裁定这是自用 CLI，不是产品。 | 不回来。要那些直接用 bit.ly。 |
| **多端同步** | 单机自用，JSON 文件足够。 | 真有多机需求时，同步 JSON 文件即可，不进工具。 |

## 留下的 MVP 边界

- 三命令：`shorten` / `expand` / `list`。
- 一个纯函数散列 + 一个 JSON 存储。
- 仅此而已。

> 注意：砍掉"删除命令"这条，间接简化了 hunt 的排查面 —— 存储只增不改，所以"短码对不上"只可能出在**生成**环节，不可能是"被删了"。范围越窄，根因空间越小。
