# 08 · ship-small：拆成能独立验证回滚的切片

> skill：`ship-small` —— 把一坨改动拆成各自能验、能上、能回滚的垂直切片。
> 触发：「/ship-small 这些改动怎么发」

ship-small 反对"攒一大坨一次推"。每个切片要能独立验证、独立回滚，且合进去不破坏主线。下面是这个工具从零到修复的切片划分。

## 切片划分

**切片 1 · 纯函数核心 + 单进程测试**

- 内容：`base62` / `urlHash` / `codeFor` + 存储读写 + 前 6 条测试。
- 独立验证：`npm test` 6 绿。
- 能回滚：纯新增，删掉即回到空。
- 价值：缩短/还原能跑通，可自用。

**切片 2 · CLI 包装**

- 内容：`cli.ts` 三命令 + `package.json` scripts。
- 独立验证：`node src/cli.ts shorten <url>` 实机跑通。
- 能回滚：只动入口层，删掉不影响核心。

**切片 3 · 确定性修复 + 跨进程回归测试**（hunt 产出）

- 内容：移除随机 salt 的那个 diff + 新增跨进程同码测试。
- 独立验证：先看回归测试**在修复前是红的**（`Soj389 != NsFeac`），打上修复后转绿（8 绿）。
- 能回滚：单点 diff，回滚即回到"跨进程漂移"的已知坏状态，边界清晰。
- 为什么单独成片：它修的是一个**正确性根因**，要让历史里这个修复点可被精确定位和回滚，不和功能开发混在一个提交里。

## 发布顺序与提交信息

```
1. feat(urlshort): 确定性短码核心 + 往返存储
2. feat(urlshort): CLI 三命令 shorten/expand/list
3. fix(urlshort): 移除散列里的随机 salt，确定性跨进程恢复 + 回归测试
```

> 切片 3 单独成片的回报在 retro 里兑现：复盘时一眼能看到"哪个提交修了确定性、之前坏了多久"。如果它和切片 1 揉成一个大提交，这条历史就糊了。
