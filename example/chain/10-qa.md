# 10 · qa：运行时验收 —— 这条 CLI 现在能不能发

> skill：`qa` —— 把已实现的功能放进真实运行时，走用户路径，用命令输出和产物检查拿证据。
> 触发：「验证 example/app」「实现完了帮我跑一遍验收」

qa 不是 review，也不是 debugging。它只回答一个问题：**这条用户路径现在能不能用**，证据来自运行时，不是读代码"觉得应该行"。代码质量交给 `check`，根因定位交给 `hunt`。下面是这条链路补上的那一环。

## 先定义验收面，再开跑

从工具的承诺反推可观察检查，而不是打开后乱点：

- happy path：`shorten` 缩短并打印短码
- 往返：`expand` 把短码还原回原 URL
- 核心前提：跨进程确定性（这正是 [06-hunt](06-hunt.md) 修的那个 bug 的属性）
- 负向 / 边界：未知短码、缺参、未知命令的退出码与 stderr
- 产物：存储落盘是合法 JSON

## 不碰用户数据

默认存储是 `app/links.json`。验收全程用 `URLSHORT_DB` 指向临时 fixture 文件，运行前先确认真实 `links.json` 不存在，不为 QA 清理或覆盖任何文件。

## 验收矩阵

```
status:       pass
scope:        example/app · urlshort 确定性短链 CLI（shorten / expand / list + 错误路径）
environment:  Node v24.16.0（README 要求 ≥23.6 ✓）
              入口 node src/cli.ts；fixture DB 经 URLSHORT_DB 指向临时文件
checks:
  - [pass] happy path: shorten https://anthropic.com -> "GY5lxK", exit 0
           （与 app/README.md 承诺值逐字一致）
  - [pass] round-trip: expand GY5lxK -> https://anthropic.com, exit 0
  - [pass] 跨进程确定性: 两个独立进程 + 各自空 DB, 同一 URL 都得 GY5lxK
           —— 证明短码生成不读存储、无运行时状态
  - [pass] list: 多条映射按短码升序、tab 分隔输出, exit 0；落盘为合法 JSON
  - [pass] 负向: expand zzzzzz -> stderr "未知短码：zzzzzz", exit 1
  - [pass] 边界: shorten 缺参 -> usage, exit 2；未知命令 -> usage, exit 2
  - [pass] 产物形状: 短码定长 6, 字母表内字符
  - [pass] 单元测试: node --test, 8/8 全绿（含"跨进程确定性"那条回归测试）
failures:     none
handoff:      none
```

## 判决：可发

重点不是单测绿，而是**用户会走的真实路径在运行时都拿到了证据**。尤其是这个工具的命根子 —— 跨进程确定性：两个互不相干的进程、各自空存储，同一 URL 仍缩出同一短码 `GY5lxK`，正是 [06-hunt](06-hunt.md) 里那个"随机 salt"真 bug 当年破坏的属性，现在扛住了。

未覆盖项与 [app/README.md](../app/README.md) 的「已知限制」一致，是文档里刻意划出的范围外项，不算缺陷：短码碰撞、自定义短码 / 删除、`expand` 信任损坏的存储文件。qa 不把范围外项写进失败清单。

## 这一环补的是什么

链路最初创建时还没有独立的 qa skill，运行时验证散在 `tdd` / `hunt` / `check` 的工件里。这篇把它收拢成一份独立的、可复跑的 CLI 验收：和 `tdd` 的"测试绿"不同，qa 给的是"真实入口 + 真实退出码 + 真实落盘"这一层证据。`tdd` 证明逻辑对，qa 证明这条命令现在能交到用户手里。
