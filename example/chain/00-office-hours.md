# 00 · office-hours：该不该做这个短链工具

> skill：`office-hours` —— 六个逼问验证一个想法值不值得做、做给谁、最窄切入点。
> 触发：「我想写个 URL 短链工具，帮我盘一盘该不该做」

office-hours 不替你拍板，它逼你把含糊的冲动顶到具体且不舒服的地方。下面是六问的纪要。

## 1. 要解决谁的什么痛？

- 含糊版："想要个短链。"
- 顶具体："我在写文档/CLI 输出里要塞长 URL，想要个**离线、可复算**的短码，不想依赖 bit.ly 这类服务，也不想要它们的跳转统计和账号。"
- 真实痛点收窄成：**离线 + 确定性**。不是和 bit.ly 抢市场。

## 2. 现状（status quo）已经怎么解决了？凭什么比它好？

- 现状：bit.ly / tinyurl（要联网、要账号、有跳转追踪）、或自建带数据库的服务。
- 我的差异点只有一个站得住：**同一个 URL 永远缩成同一个短码，可在任意机器纯函数复算**，不需要中心计数器或数据库自增 ID。
- 如果丢掉"确定性"这条，这工具就毫无存在必要 —— 这是整个想法的命根子。

## 3. 真有人要吗？还是你自己想写？

- 诚实回答：主要是自己要，需求量小。
- 结论修正：这是个**工具自用**项目，不是产品。那就别按产品标准投入，按"够用且别坑自己"投入。

## 4. 最窄能验证的切入点是什么？

- 不做账号、不做跳转、不做 Web 服务、不做统计。
- 最窄切入点：一个本地 CLI，三条命令 `shorten / expand / list`，一个 JSON 文件存储。

## 5. 什么情况下你会后悔做了它？

- 如果它的短码**不稳定**（今天缩的明天对不上），那它比直接贴长 URL 还坑 —— 因为我会以为短码是可信的。
- 记下来：确定性一旦破，这工具的价值是负的。（这条后来在 hunt 里真应验了。）

## 6. 做 vs 不做，各自的代价？

- 做：半天写完，自用，维护成本近零。
- 不做：继续贴长 URL，能忍。
- 裁决：**做，但锁死在最窄 MVP**。它的唯一卖点是确定性，所以接下来 think 要把"确定性"作为不可让步的设计前提。
