# 同步已有：逐文件审查启发式

逐个读文档文件，与本回合的 diff 交叉核对，把过时的改对。这些是通用启发式，适配你所在的任何项目，不限某个技术栈。

每个文件把需要的改动分两类：

- **Auto-update（直接 Edit）**: diff 清楚支撑的事实修正：表格加一项、改文件路径、改计数、更新项目结构树、修陈旧交叉引用。
- **Ask user（停下问）**: 叙事改写、删整节、安全模型变更、大段重写（单节超 ~10 行）、相关性不明、新增整节。

## README.md

- diff 里出现的所有功能/能力，README 都描述到了吗？
- 安装/配置步骤和改动一致吗？
- 示例、demo、用法描述还成立吗？
- 排障步骤还准吗？

不自动改：README 的介绍段、项目定位。这些要问。

## ARCHITECTURE.md

- ASCII 图、组件描述和当前代码对得上吗？
- 设计决策、"为什么这么做"还准确吗？

保守。只改 diff 明确推翻的部分。架构文档描述的是不常变的东西，设计理由不自动改。

## CONTRIBUTING.md（新贡献者冒烟测试）

- 把自己当全新贡献者走一遍配置步骤。
- 列出的命令准吗？每一步会成功吗？
- 测试分层描述和当前测试基建一致吗？
- 开发配置、工作流描述还是当前的吗？
- 标出任何会让首次贡献者失败或困惑的地方。

## 项目说明文件（CLAUDE.md / AGENTS.md / 等）

- 项目结构小节和真实文件树一致吗？
- 列的命令和脚本准吗？
- build/test 指令和 package.json（或等价文件）一致吗？

## 其它 .md

读文件，判断它的用途和受众，再与 diff 交叉核对有无矛盾。

## CHANGELOG.md（关键：只润色，永不覆盖）

CHANGELOG 的每条都是历史。这一步只调整语气措辞，不重写、不替换、不重新生成。

规则：

1. 先读完整个 CHANGELOG，搞清已有什么。
2. 只在已有条目内部改措辞。永不删、重排、替换条目。
3. 永不从头重新生成某条。条目是某次 ship 从真实 diff 写的，是真相源。
4. 条目看着错或不全，用提问确认，不要静默改。
5. 用 Edit 精确匹配，永不用 Write 覆盖 CHANGELOG.md。

CHANGELOG 在本分支没改动就跳过这步。

改了就过 voice：

- 卖点测试：用户读每条会想"哦不错，我想试试"吗？不会就改措辞（不改内容）。
- 先说用户现在能做什么，不是实现细节。"你现在可以…"而不是"重构了…"。
- 读起来像 commit message 的条目，改写。
- 内部/贡献者向的改动，归到单独的"### For contributors"小节。
- 小的语气调整直接改，会改变含义的重写先问。

## 跨文档一致性 & 可发现性

逐文件审完做一遍跨文档检查：

1. README 的功能列表和项目说明文件描述的一致吗？
2. ARCHITECTURE 的组件列表和 CONTRIBUTING 的项目结构描述一致吗？
3. CHANGELOG 最新版本和 VERSION 文件一致吗？
4. **可发现性**: 每个文档文件都能从 README 或项目说明文件被链接到吗？有 ARCHITECTURE.md 但两个入口都不链接它，标出来。
5. 矛盾：明显事实不一致（如版本号对不上）自动修，叙事矛盾用提问。

## VERSION（永不静默 bump）

- 没有 VERSION 文件：跳过。
- 检查本分支是否已改 VERSION：`git diff <base>...HEAD -- VERSION`。
- 没 bump：停下问（推荐 Skip，纯文档改动通常不需要 bump；或 patch / minor）。
- 已 bump：不要静默跳过。读当前版本的 CHANGELOG 条目，对照完整 diff，看 bump 是否覆盖了本分支全部改动。有重大未覆盖改动就问，要不要给它单独一个版本。
