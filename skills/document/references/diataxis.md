# 生成缺失：Diataxis 四象限

Diataxis 把技术文档分成四类，按两个轴划分：服务"学习 vs 工作"，面向"实践步骤 vs 理论知识"。混在一起写是文档难用的根因（教程里塞 API 参数，参考手册里夹哲学）。

```
              实践导向(action)        理论导向(cognition)
          ┌──────────────────────┬──────────────────────┐
学习时    │  Tutorial            │  Explanation         │
(study)   │  教程：手把手带初学  │  解释：讲清为什么、  │
          │  者完成第一次成功    │  背景、设计取舍      │
          ├──────────────────────┼──────────────────────┤
工作时    │  How-to              │  Reference           │
(work)    │  操作指南：解决一个  │  参考：精确、完整、  │
          │  具体目标的步骤      │  可查的事实清单      │
          └──────────────────────┴──────────────────────┘
```

## 四类的判别问题

- **Tutorial（教程）**: 一个全新用户跟着走，能不能从零到第一次跑通？面向学习，重在"被带着成功一次"，不求完整覆盖。
- **How-to（操作指南）**: 已经会用的人想完成某个具体任务（部署、接某个 API、配某个功能），有没有针对性的步骤？面向工作，目标驱动，假设读者已有基础。
- **Reference（参考）**: 想查某个命令/函数/配置项的确切签名、参数、返回值，有没有一处完整准确的清单？面向工作，重准确与完整，不解释、不教学。
- **Explanation（解释）**: 想搞懂某个设计为什么这么做、背景、权衡，有没有讲清来龙去脉的文字？面向学习，重理解，不是操作步骤。

## 怎么用它找缺口

不是四象限都写满才叫完整。流程：

1. 把现有 `.md` 文件逐个归到某个象限（README 常混了多类，按其主体归）。
2. 看哪个象限是空的或薄的，结合本次 diff 新增了什么能力，判断用户最缺哪一类。
3. 只补最该补的那类。一个新功能 ship 了但只有 README 一句话，最常缺的是 How-to（怎么用）和 Reference（参数表）。
4. 宁可少而准。没人会看的文档不如不写。

## 各类的结构模板

**Tutorial**

- 一句话说明跟完能做到什么 + 前置条件
- 编号步骤，每步一个可见结果，从零到第一次成功
- 全程用具体的、能复制粘贴的命令/代码
- 不岔开讲选项和原理，链接到对应 How-to / Reference

**How-to**

- 标题就是要解决的目标（"如何部署到 X"）
- 假设读者已有基础，直接给步骤
- 聚焦这一个任务，不教学、不铺垫
- 列出前提和可能的坑

**Reference**

- 按可查的结构组织（按命令、按配置项、按 API 端点）
- 每项：签名 / 参数 / 类型 / 默认值 / 返回 / 示例
- 准确完整，不夹叙事、不讲为什么
- 与代码对齐：参数和实际签名一致（从 diff/源码核对）

**Explanation**

- 讲背景、动机、设计取舍、为什么不选别的方案
- 面向理解，可以有观点
- 不放操作步骤，链接到 How-to / Tutorial

## 生成后

新文档用 Write 创建，放在合适位置（常见 `docs/` 或仓库根）。建完务必检查能从 README 或项目说明文件被链接到，否则等于没写。每个新建文件在最终健康摘要里标 `Created（补的是哪个象限缺口）`。
