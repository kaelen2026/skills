# Observe: 让线上系统自己告诉你它出事了

一个人开发没有 on-call 团队轮班盯屏，所以更不能靠用户来通知你挂了。失败模式有两个相反的方向：一是裸奔，出了事是用户截图来骂你才知道，没有任何信号能回答"现在是不是挂了、从什么时候、影响谁"；二是过度埋点，仪表盘密密麻麻没人看，告警天天误报被你静音，等真出事那条也一起被忽略了。

**从用户能感知的失败倒推要埋什么，不是有什么指标就埋什么。** 先保证能回答三个问题：可用吗（是否在线）、快吗（延迟）、有没有在报错（错误率与堆栈）。这三类信号 + 一个主动推到你手机的告警，比一屏花哨 dashboard 有用得多。

## Outcome Contract

- Outcome: 一份最小可观测方案，让线上故障在用户发现之前主动找到你，且不被噪音淹没。
- Done when: 至少有一个外部黑盒探活、错误聚合上报、能串起一次请求的结构化日志；每个告警都对应一个你会真的起身处理的动作；日志里没有 PII / 密钥；采样与上报量在预算内。
- Evidence: 现在有没有任何监控、用户目前怎么发现出事、部署形态（单机 / 容器 / serverless）、预算、有没有现成平台（Sentry、平台自带 metrics、uptime 服务）。
- Output: 埋哪三类信号、各回答什么问题、告警阈值与推送渠道、隐私与成本约束、第一个该加的探活。

## Core Stance

- **先回答"是不是挂了"，再谈别的。** 在埋任何业务指标之前，先有一个能告诉你"系统整体现在健康吗、从什么时候不健康"的信号。其余都建立在这之上。
- **告警要主动推送，且必须可执行。** 一个人不会一直看屏，告警得推到手机/IM。但每条告警都要对应"收到后你会做的一件具体事"，否则它是噪音，迟早被静音，连带真问题一起被忽略。
- **黑盒探活独立于应用。** 至少一个从外部打你服务的探活（uptime ping）。只靠应用内自报，应用挂了它也不会报。
- **错误聚合，不要逐条告警。** 同一个异常聚合成一条带堆栈和上下文的报告（Sentry 类），按"新出现 / 频率突增"提醒，而不是每个异常发一条，制造告警风暴。
- **日志能串起一次请求。** 结构化 + 关联 id（request id / trace id / user id），出事时能捞出"这一次具体请求经过了哪些步骤"，而不是一堆对不上的散字符串。

## Pre-flight

- 先认清现状：现在有没有任何监控？上一次线上出事你是怎么知道的（用户报 / 自己撞见 / 至今不知道）？这决定从零起步还是反向砍噪音。
- 弄清部署形态：单机 / 容器 / serverless，决定日志往哪收、探活怎么打、指标从哪来。
- 看预算与现成平台：有没有在用 Sentry、平台自带 metrics/logs、免费 uptime 服务。优先用现成的，别为可观测自建一套要你维护的系统。

## 模式

**From Zero · Activate when**: 现在基本裸奔，没有监控，靠用户发现问题。

按性价比上最小三件套，先有再优化：

1. 外部 uptime 探活：一个免费服务定时打你的健康端点，挂了推送到手机。这是"是不是挂了"的兜底。
2. 错误聚合上报：接一个 Sentry 类服务，未捕获异常自动带堆栈和上下文上报、去重。
3. 一条能串请求的结构化日志：给每个请求一个 id，关键路径打结构化日志，出事能捞出整条链路。
这三个到位，才考虑业务指标和 dashboard。

**Tame the Noise · Activate when**: 已经有一堆告警/监控，但误报多、告警疲劳、该看的没人看。

反过来做减法：把每条告警过一遍"收到后我会做什么"，答不出具体动作的，从告警降级为 dashboard 或日志。告警阈值改成基于用户影响（错误率、p99 延迟、探活失败）而非机器指标抖动（CPU 瞬时高）。留下少数几条真正可执行、低误报的，让它们重新可信。

## Hard Rules

- **每条告警必须对应一个你会真的起身处理的动作，阈值基于用户影响。** 做不到就不是告警，是噪音；它会被静音，并把真问题一起带走。
- **不许把 PII / 密钥 / token 写进日志或错误上报。** 上报前脱敏，日志里不落明文凭证和个人数据。
- **加任何指标前先写它要回答的问题。** 谁在什么时候看、看了做什么决定？答不出就别加，避免堆出没人看的仪表盘。
- **至少一个独立于应用的外部探活。** 只靠应用内自报，进程挂了、机器宕了就没有信号。
- **错误上报去重聚合。** 同类异常合并成一条，按新增或突增提醒，不要逐条轰炸。
- **日志要能定位一次具体请求。** 没有关联 id 的散日志，线上只能得出"大概那会儿有人报错"，定位不到根因（那是 hunt 的入口，但 hunt 需要这里先埋好证据）。
- **采样与留存看预算。** 全量 trace、无限留存会烧钱。高频路径采样，按价值定留存期。

## Gotchas

| 出过的问题 | 规则 |
| --- | --- |
| 线上挂了几小时，用户截图来骂才知道 | 先上外部 uptime 探活，主动推送到手机 |
| 只有应用内健康检查，进程崩了它也不上报 | 探活必须从外部独立打 |
| 告警天天误报，最后全静音，真出事也没看 | 每条告警对应可执行动作，阈值基于用户影响 |
| 异常逐条发告警，一次故障刷出几百条 | 聚合去重，按新增/突增提醒 |
| 日志是散字符串，出事捞不出整条请求 | 结构化 + 关联 id，能串起单次请求 |
| 把用户手机号、token 打进日志 | 上报前脱敏，凭证和 PII 不落日志 |
| 埋了一屏指标，没人看也没驱动任何决定 | 加指标前先写它要回答谁的什么问题 |
| 开了全量 trace，月底账单爆炸 | 高频路径采样，按价值定留存 |

## Output

一份最小可观测方案：

```
可观测方案

现状: [有无监控 / 用户目前怎么发现出事 / 部署形态 / 预算 / 现成平台]

信号（三类，各回答一个问题）
- 可用性: [外部探活方式] → 回答"现在是不是挂了、从何时起"
- 错误:   [错误聚合上报] → 回答"在报什么错、堆栈、影响范围"
- 性能/链路: [结构化日志 + 关联 id / 关键延迟指标] → 回答"哪一次请求慢/错在哪一步"

告警（每条都可执行）
- [触发条件，基于用户影响] → 推送到 [渠道] → 收到后做 [具体动作]

约束
- 隐私: 不落 [PII / 密钥] 清单
- 成本: 采样率 / 留存期

第一步: [当前最该补的那一个信号，通常是外部探活]
```

收尾这句：

```
可观测方案已就绪。先上第一步那个信号让线上不再裸奔，其余按性价比逐个补。线上真出事时，带着这里埋下的证据走 /hunt 定位根因。
```
