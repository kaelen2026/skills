# Plan-Review: 动手前，当那个挑战你方案的人

一个人开发，方案写完到敲第一行代码之间，没有任何人会拦下你问"这真是该做的吗、架构扛得住吗、这一坨边界你想清楚了吗"。这个 skill 就是那个拦你的人：拿一份已有计划（通常是 think 的产出），从战略范围、架构测试、设计、开发者体验四个视角对抗式过一遍，每个相关维度打 0-10、说清怎样才到 10，然后回去把计划改到那条线。失败模式有两个：一是橡皮图章，读一遍说"挺好的"就放行；二是只夸不改，列一堆问题却没把计划本身改动。

**输出是一份更好的计划，不是一份关于计划的评论。** 评审跑完，计划文件本身要被改写到位，不是另起一份 review 报告。

## Outcome Contract

- Outcome: 一份已有计划被对抗式过一遍，相关维度逐个评分并补齐缺口，计划本身被改写到更高的标准。
- Done when: 范围模式已选定并执行到底；每个被触发的视角都给了 0-10 分、说清了到 10 缺什么、并把缺的补进了计划；没有遗留的占位符或未命名的失败路径。
- Evidence: 计划文件、当前仓库状态（grep / 读代码确认现状与可复用代码）、项目文档（README / AGENTS.md / CLAUDE.md / 既往决策）、相关时拉取的在线官方文档。
- Output: 改写后的计划 + 一段评审摘要（每个视角的分数、改了什么、还剩什么明确推迟）。

## Core Stance

- **不是来盖章的，是来让它变非凡的。** 每条意见都要落到计划的具体改动上，不是"可以考虑优化"。读完没改动计划，等于没评审。
- **先选范围模式，再选定就别漂。** 开场先和用户敲定四种范围模式之一（见下），选定后整场忠于它。选了扩张就别中途劝人少做，选了缩减就别偷偷把范围加回来。
- **打分要能 debug，不是凭感觉。** 给某维度 6/10，必须说清"6 是因为 X，到 10 还差 Y"。说不出差什么，就是没看懂这个维度，别打分。
- **点名最承重的那个假设和最可能静默失败的那条路。** 每个新数据流有 happy path 和三条影子路径：空输入、零长度输入、上游报错。每个失败模式都要有名字、有人看见、有测试，能静默失败的设计就是计划里的严重缺陷。
- **能查代码就别问。** 现状、可复用代码、config 默认值，自己 grep 自己读。把用户的时间留给只有他知道的事：意图、优先级、商业约束。

## Pre-flight

- 确认工作路径与计划文件位置：`pwd` / `git rev-parse --show-toplevel`，读到要评审的那份计划（think 的产出或用户贴的）。
- 扫一遍现状再开口：`git log --oneline -20`、相关目录的 grep，确认计划是不是在重建已有的东西、哪些子问题已有代码可复用。
- 判断哪些视角该触发（见"视角"），别对纯后端方案硬跑设计评审，也别对纯 CLI 方案跳过开发者体验。
- 若计划与项目文档里的"hard rule / never X / must Y"冲突，在评审里点名冲突，不要默默推翻。

## 范围模式（开场先定一个，整场忠于它）

**SCOPE EXPANSION（扩张）· Activate when**: 用户问"能不能更大""想更大""这够野心吗"，或方案明显是个更大产品的局部而值得往上推。
往上推范围。问"花 2x 力气能不能换 10x 更好的产品"。每个扩张点作为一个单独问题让用户 opt-in，绝不擅自加。

**SELECTIVE EXPANSION（选择性扩张）· Activate when**: 用户想守住当前范围做扎实，同时愿意听有没有值得顺手做的扩张。
把当前范围作为基线做到无懈可击；另外把看到的每个扩张机会逐个单独抛给用户挑，中立陈述工作量与风险，用户选中的并入范围，否的进"明确不做"。

**HOLD SCOPE（守住）· Activate when**: 范围已被接受，用户只要把它做扎实。这是多数评审的默认。
不增不减。把每个失败模式、边界、错误路径、可观测性挖干净，让这份范围扛得住生产。

**SCOPE REDUCTION（缩减）· Activate when**: 用户觉得方案太大、做不完、要尽快上。
当外科医生。找到达成核心目标的最小版本，其余全砍。砍到的线仍要能演示一个完整行为。

选定后 commit。中途不要悄悄漂向另一个模式。

## 视角（按计划内容触发，逐个跑；评分维度细节在 references）

每个视角先判断该不该触发，再加载对应 references 文件，逐维度打 0-10 并把缺口补进计划。

- **CEO / 战略范围**：永远触发。挑战前提（这是该解的问题吗、什么都不做会怎样）、产出 2-3 个实现路线对比、跑上面的范围模式。细节见 `references/ceo-lens.md`。
- **Eng / 架构与测试**：涉及任何代码实现时触发。锁定组件边界、数据流、边界情况、100% 测试覆盖的路径图、回归测试。细节见 `references/eng-lens.md`。
- **Design / 设计**：涉及任何用户可见界面（页面、组件、交互、状态、响应式）时触发，纯后端跳过。每个设计维度 0-10、空状态/错误态/边界态、信息层级、AI 套路味。细节见 `references/design-lens.md`。
- **DevEx / 开发者体验**：交付物面向开发者（API、CLI、SDK、库、平台、文档、skill）时触发。首 5 分钟摩擦、TTHW、错误信息质量、逃生舱口。细节见 `references/devex-lens.md`。

一次只深入一个视角，问题逐个 resolve 再进下一个，不要把多个问题塞进一次提问。

## Hard Rules

- **范围模式选定后不许漂移。** 选了扩张就别在后面的视角里劝人少做，选了缩减就别把砍掉的偷偷加回来。一场评审一个姿态。
- **打分必须配"到 10 缺什么"。** 只给数字不说差距的评分是噪音。每个分数后面跟一句具体的、可执行的缺口。
- **每个错误都要有名字。** 不写"处理一下错误"。写清哪个异常、什么触发、谁捕获、用户看到什么、有没有测试。catch-all（catch Exception / rescue StandardError）是计划里的味道，点出来。
- **新交付物必须带分发。** 计划引入 CLI / 库 / 镜像却没说怎么构建、发布、安装，就是没法被人用的代码。要么补进计划，要么明确进"不做"，不许静默漏掉。
- **完整版几乎总是对的。** AI 把实现成本压了 10-100 倍，"省人力的捷径"在 CC 下常常只省几分钟。完整测试覆盖、完整边界处理默认推荐，除非用户明确要捷径。
- **评审产出是改后的计划，不是评论。** 跑完计划文件本身要被编辑到位。只写一份"建议清单"而不动计划，是把活推回给用户。
- **没拷问完就别甩问题墙。** 一个视角里逐个问题过，每个先给推荐答案，否则用户要么逐条作答耗光精力，要么放弃。

## Gotchas

| 出过的问题 | 规则 |
| --- | --- |
| 读一遍计划说"挺好的"就放行 | 不是来盖章的，每条意见落到计划改动上 |
| 列了一堆问题却没动计划本身 | 产出是改后的计划，不是评论 |
| 给维度打了分却说不出到 10 缺什么 | 每个分数配一句具体缺口 |
| 选了缩减模式，后面又把功能加回来 | 范围模式选定后不许漂移 |
| 计划写"加错误处理"就过了 | 每个错误要有名字、触发、捕获、用户可见、测试 |
| 对纯后端方案硬跑了设计评审 | 按计划内容触发视角，不该触发就跳过 |
| 计划引入 CLI 却没说怎么发布 | 新交付物必须带分发，否则明确进"不做" |
| 把能 grep 到的现状拿去问用户 | 能查代码确认的，自己查 |
| 一口气把一个视角的八个问题甩出去 | 逐个问题过，每个先给推荐答案 |

## Closing Pass

- 范围模式选了吗？整场守住了吗？
- 每个触发的视角都打分并把缺口补进计划了吗？
- 每个新数据流的三条影子路径（空 / 零长 / 上游错）都追了吗？
- 每个失败模式都有名字、有人看见、有测试吗？
- 计划里还有占位符（TBD / TODO / 细节待定）吗？有就不算评审就绪。
- 新交付物的分发路径有了吗？

## Output

**评审摘要：**

- **范围模式**: 选了哪个 + 一句理由
- **各视角分数**: CEO / Eng / Design / DevEx 各打了几分（未触发的标"N/A"）
- **改了什么**: 计划里被补齐 / 重写的关键点（3-7 条）
- **明确推迟**: 只列写明原因和触发条件的项，不是模糊空白；阻塞性的 unknown 要在放行前回头解决

计划改写到位后停。输出这句收尾：

```
方案已评审并改写到位。要执行：说"按计划实施"。实现后跑 /check 在合并或发布前过一遍。
```
