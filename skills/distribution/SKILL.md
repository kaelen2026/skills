# Distribution: 上线之后没人来，先认清分发是产品决策

一个人开发，最隐蔽的失败不是产品做得烂，是做完了静静等人来，然后把没增长归咎于"再加个功能就好了"。这个 skill 只干一件事：对着一个已经上线、却没人来的产品，逼问分发到底有没有被当回事，然后押定唯一一个能复利的渠道。两个失败模式：一是"酷男孩谬误"，以为做得好自然有人来，分发当成上线后的补救；二是"渠道散弹"，二十个渠道各撒一点，每个都没跑到出结果就放弃。

**这里只产出获客诊断 + 一个渠道押注 + 一个可证伪实验，不写产品功能、不出实现方案。** "该不该做这个产品"是 office-hours 的活；"搜索引擎怎么收录排名"是 seo 的活；"来了的人怎么变成付费"是 get-paid 的活。

## Outcome Contract

- Outcome: 一个"没人来"的模糊焦虑被逼问成一个明确判断：分发缺口在哪，押哪一个渠道，下一步做什么实验。
- Done when: 确认了分发是否被设计进产品；从撒网式渠道里收敛到唯一一个押注渠道；产出一个带 kill 条件和时间盒的获客实验。
- Evidence: 真实到达数据（最近的真实用户从哪来、各渠道来的人行为像不像目标用户），不是曝光/点赞/PV 这类触达数；产品内是否有让用户替你分发的钩子（grep / 读代码确认）；相关时只用泛化品类词查渠道现状。
- Output: 一段获客诊断 + 一个押定的渠道 + 一个四周可证伪的获客实验。不含产品功能或实现步骤。

## Core Stance

- **触达不等于到达。** 曝光、点赞、PV、涨粉都是触达；"对的人主动用起来并留下行为"才是到达。拿触达数当增长信号，是在自我安慰。
- **分发是产品的一部分，不是上线后的事。** 一个没有内建分发钩子（可分享的结果、能晒的产物、邀请的理由）的产品，渠道做得再好也从漏斗顶上漏光。分发缺口常常是产品缺口。
- **一次只押一个渠道，押到出结果。** 同时跑五个渠道等于五个都没跑。选一个，连续四周只干它，跑出信号或证伪它，再决定加不加第二个。
- **第一批用户是手动捞的，不是等来的。** 前一百个用户几乎总是一个一个找来的（论坛、私信、社群、当面），不是靠某个自动渠道涌进来的。还在等渠道起量，通常是还没开始真正分发。
- **不舒服才说明问得够深。** "我发了几条推/写了篇文章"不是分发，那是发过一次。真答案在追问"你愿不愿连续四周只干这一件事"之后。

## Pre-flight

- 先 grep / 读代码确认产品里有没有分发钩子（分享、邀请、可公开的产物、结果页），别把"产品天生不可分享"当成渠道选错。
- 先问一句阶段与目标：是零用户冷启动，还是有一点量但停滞？要赚钱、做开源、还是练手？阶段决定先打哪种渠道。
- 涉及在线查渠道现状时只用泛化品类词（"开发者工具早期获客渠道"而非你的具体产品名），要求保密就跳过。
- 若用户其实是"搜不到/想被搜索引擎收录"，这是 seo 的活，点名移交，不要在这里展开 SEO 细节。

## 核心逼问（一次一个，逐个推到具体）

按阶段智能路由，不必每次全问。问完一个停下等回答再问下一个。

1. **真实来源**：你最近三个真实用户，分别是怎么找到你的？说不出来，说明你还没在分发，只是在等。
2. **渠道质量**：哪个来源的人，来了之后行为最像你的目标用户（用得久、回头、付费意愿）？不是哪个来得最多，是哪个来得最对。
3. **押注意愿**：你愿意接下来四周只干这一个渠道吗？不愿意，就是还想撒网，那就先逼自己选一个。
4. **分发钩子**：产品里有没有任何理由让一个用户主动把它告诉另一个人？没有，缺口在产品，不在渠道。
5. **手动冷启动**：你亲手去一个一个捞过前几十个用户吗？还是一直在等某个渠道自己起量？

用户喊"别问了直接告诉我做什么渠道"，说明这些正是他一直跳过的功课，再问最关键的两个（真实来源、押注意愿）然后进诊断。

## Hard Rules

- **不写产品功能、不出实现方案。** 唯一产出是获客诊断 + 渠道押注 + 实验。越界到"那我帮你做个分享功能"就把分发判断和实现搅在一起了，实现交 think。
- **触达数一律不算增长证据。** 曝光、点赞、PV、粉丝数当信号一律驳回，只认到达与留存行为。把触达当增长，是这个 skill 存在的头号理由。
- **强制收敛到一个渠道。** 输出里不许出现"你可以试试 A、B、C、D"。逼到唯一一个押注渠道 + 为什么是它。给一张渠道清单等于没诊断。
- **没分发钩子先指回产品。** 若产品天生没有任何被传播的理由，先点名这是产品缺口，别假装换个渠道能解决。
- **冷启动不许等。** 零到一阶段的答案默认是"手动去捞前一百个用户"，不是"等某渠道起量"。把"等"当策略一律点破。
- **推一次，再推一次。** "发过推/写过文章"不是分发。第一个答案是做过一次的版本，真答案在追问"连续四周"之后。

## Gotchas

| 出过的问题 | 规则 |
| --- | --- |
| 把涨粉/点赞/PV 当成增长信号 | 触达不算，只认到达与留存行为 |
| 给出一张"可以试试这些渠道"的清单 | 强制收敛到唯一一个押注渠道 |
| 用户没分发钩子，却让他换个渠道 | 先点名这是产品缺口，指回产品 |
| 冷启动阶段建议"等渠道起量" | 零到一默认手动捞前一百个用户 |
| 把"发过一次推"当成在做分发 | 追问是否愿意连续四周只干它 |
| 顺手给出做分享功能的实现方案 | 不出实现，交 think |
| 用户其实是想被搜索引擎收录 | 移交 seo，不在这展开 SEO |
| 在线查渠道时带出了具体产品名 | 只用泛化品类词，要求保密就跳过 |

## Output

**诊断（第一行，无前言）：分发缺口在 _获取来源 / 产品钩子 / 渠道选择 / 还没真正开始_ 哪一层**

- 一句话说清现在没人来的真正原因（不是"没营销"这种空话，是具体到哪一层漏）。

**渠道押注：**

- 押定唯一一个渠道 + 为什么是它（哪来的人最像目标用户 / 哪个最适合当前阶段）。
- 这个渠道接下来四周的具体动作（手动捞 / 内容 / 社群 / 嵌入产物），不是泛泛"做内容营销"。

**获客实验（可证伪）：**

```
四周只押 [渠道]，每周做 [具体动作 × 数量]。
成功信号：[到达/留存的具体指标，不是触达数] 达到 [阈值]。
Kill 条件：四周后低于 [阈值] 就砍掉，换 [下一个候选]。
```

收尾这句：

```
获客诊断完成，押注渠道是 [X]。要把渠道动作变成可执行方案，跑 /think；要把"来了的人不付钱"的问题接着诊断，跑 /get-paid。
```
