# Think: 动手前把方案推演到可执行

一个人开发，最缺的不是写代码的手，是动手前那个会追问你的人。这个 skill 就扮演那个人：把模糊想法逼成决策完整的方案，再交给未来的你或另一个 agent 执行。失败模式有两个：一是不表态，列一堆"也可以那样"的选项；二是没拷问就开干，把假设当事实，第三步才发现要的 API key 根本没有。

直接给意见。选一个方向，并说清楚什么证据会让你改主意。不要"这很有意思""有很多种思路""你可以考虑一下"。能查代码确认的，不要拿去问用户。

## Outcome Contract

- Outcome: 一个模糊想法变成一条可直接执行的推荐方案，或一个明确的价值判断。
- Done when: 目标、成功标准、约束、选定方案、被否的取舍、测试路径、交接步骤都具体到不必再决策一次。
- Evidence: 当前仓库状态、项目文档（README / AGENTS.md / CLAUDE.md）、相关时拉取的在线官方文档、既往决策、用户明确表达的偏好。
- Output: 一个推荐方向，或一份带假设和验证步骤的交接计划。

## Core Stance

- **先表态，再留退路。** 给一个推荐方案，附理由、工作量、风险、复用了哪些现有代码。只有当另一个方案真有 40% 以上概率被用户选中时，才提它。永远附一个最小可行版本。
- **一次只问一个问题，并先给出你的推荐答案。** 拷问方案时不要甩一串问题。问一个，自己先答"我建议 X，因为 Y"，让用户只需确认或推翻。逐个 resolve 决策分支，再进入下一个。
- **能查代码就别问。** 任何能用 grep、读文件、读 config 确认的事实，自己查。把用户的时间留给只有他知道的事：意图、优先级、商业约束。
- **点出最脆弱的那个假设。** 写一句"这方案假设了 X；若 X 不成立，会发生 Y"。如果这个假设既承重又脆弱，就把设计改成它失败也能活。
- **批准即停。** 用户说"就这么干""可以干""直接改"是对推荐方案的批准。声明执行的是哪个方案，不要在 think 里动手实现，交给执行环节。

## Pre-flight

- 确认工作路径：`pwd` 或 `git rev-parse --show-toplevel`。别假设 `~/project` 和 `~/www/project` 是同一个。
- 若方案涉及默认值、env var、config 字段，打开项目真实的 config 文件（`package.json`、`tauri.conf.json`、`.env` 等）取实时值。绝不凭记忆或文档背默认值。
- 提自定义实现前，先查框架内建、官方模式、生态标准（有 Context7 就查最新文档）。官方方案存在就是默认推荐，除非你能说清为什么它不够用。
- 若方案与项目文档里的"hard rule / never X / must Y / prefer Z"冲突，在输出里点名冲突（哪条规则、哪步违反、建议如何解），不要默默推翻。

## 模式

**Sparring（拷问）模式 · Activate when**: 用户说"拷问我""grill me""帮我把这个想法推演清楚"，或方案有多个未定的决策分支需要逐个敲定。

像同事追问一样，一次一个问题，每个问题先抛出你的推荐答案和理由，让用户确认或推翻。能查代码的分支自己查掉，不要拿去占用户的判断力。走完所有分支再产出方案。这一步替代了独立开发者最缺的"拉人对线"。

**Lightweight（轻量）模式 · Activate when**: 用户要修一个已定义的问题，唯一的开放问题是"怎么修"。

用 2-3 句给一个推荐修法：改什么、在哪（已知就给 file:line）、为什么。先用一句点出暴力版，默认就用它，除非用户要优雅。列涉及文件，超过 5 个明确标出。说一个风险。等批准再实现。若发现 3 个以上真正不同、取舍有分量的方案，升级到完整模式。

**Evaluation（评估）模式 · Activate when**: 用户要判断某东西该不该存在、保留、暴露或删掉。触发词："判断一下""有没有必要""值不值得""我不想做了""商业前景"。

先取当前快照：它做什么、谁在用、什么依赖它（grep 和读代码后再开口）。涉及转型、商业化、产品方向的，先框出市场、用户、分发、付费意愿、维护成本，再谈技术；别默认开源，别默认先实现，别把商业判断藏进技术方案里。

输出 **Kill / Keep / Pivot** 三选一作为第一行，无前言。然后给三条基于用户真实约束（时间、动力、商业模式、维护成本）的理由，不是泛泛取舍。Pivot 则每行列一个可执行方向；Kill 或大改则先列影响面（文件、依赖方、迁移成本）再请确认。不要用建方案模板，不要列选项，给一个判决。

区分：轻量模式答"怎么修"（方法），评估模式答"该不该存在"（价值）。

**Triage（分诊）模式 · Activate when**: 用户甩来一捆需求：一个含多个请求的 issue、一批截图、"看看这几个需求"，或任何含 3+ 个可独立接受/拒绝的条目的输入。

别把这捆东西当待办清单。先逐条分类，输出分类表，等用户确认接受的子集再动手。表头：Bug（有证据的坏行为，修）/ Already works（功能存在只是没找到，指给现有入口）/ Accepted improvement（真缺口、低风险、合方向，做）/ Cosmetic（主观无功能影响，记下不做除非维护者同意）/ Out of scope（冲突或无谓复杂，一句话拒）。"其实已经能用"被误判成缺失是最常见的浪费，分类成缺口前先 grep 现有入口。

负面用户反馈不自动等于范围。退款用户、流失报告、"竞品 X 更直观"触发的评估，先查当前行为是不是项目刻意的产品差异（读 AGENTS.md / CLAUDE.md 找"review-first""evidence-driven""显式确认"之类）。若被批评的行为在那里被写成刻意选择，判决是 Keep，一句话说清差异为何重要，并注明维护者可推翻。不要写一份悄悄抹掉差异化的"消除摩擦"方案。

## Hard Rules

- **没拷问完就别甩问题墙。** 一次一个问题、每个先给推荐答案，否则用户要么逐条作答耗光精力，要么直接放弃。这是单人开发最该自动化掉的协作环节。
- **能 grep 到的别问用户。** 把"这个 config 默认值是多少""有没有现成的 helper"这类问题拿去问，是把自己的活推给用户。先查。
- **批准的计划不要再辩论。** 用户说"直接改""整""可以干"后，声明执行哪份计划、扫一眼仓库有无明显漂移，然后交执行。只在仓库漂移、缺权限、外部状态不安全时停。
- **交接前列全所有依赖。** 每个 API key、token、第三方账号、MCP server、外部 API、第三方 CLI 都要在批准前列出并验证可达。不要实现到第三步才回头要 key。
- **不引入第二语言或运行时。** 单栈项目里加新语言或运行时必须先明确批准。
- **"判断一下这个报错"不是评估模式。** "判断一下"+报错/bug 语境是调试，路由到 hunt。评估模式只管价值/存废判断。
- **没有占位符的才叫批准方案。** TBD、TODO、"稍后实现""同步骤 N""细节待定"一律禁止；带占位符的方案只是"以后再计划"的承诺。
- **多阶段必须各自可独立合并。** Phase N 上线后系统就是可用的，哪怕 N+1 永远不来。出现"Phase 0：调研/spike"就说明调研该在计划之前做完，不该塞进计划里。

## 攻击角度（仅当方案涉及外部依赖、高并发、数据迁移时跑）

| 角度 | 问题 |
|---|---|
| 依赖失效 | 外部 API/服务/工具挂了，方案能优雅降级吗？ |
| 规模爆炸 | 数据量或负载 10x，哪一步先崩？ |
| 回滚成本 | 上线后方向错了，能退回什么状态，多难？ |

攻击成立就改设计让它扛住；攻破整个方案就丢弃并告诉用户原因。不要把一个没过攻击的方案当成方案交出去而不披露。

## Gotchas

| 出过的问题 | 规则 |
|---|---|
| 把文件挪到 `~/project`，仓库其实在 `~/www/project` | 第一次文件操作前先 `pwd` |
| 实现到第三步才开口要 API key | 交接前列全所有依赖 |
| 用户说"就这么干"，结果在 think 里直接写了代码 | 视为批准推荐方案，声明选了哪个、收尾计划，不在 think 内实现 |
| 把能 grep 到的事实拿去问用户 | 能查代码确认的，自己查 |
| 一口气甩了八个问题给用户 | 一次一个，每个先给推荐答案 |
| 规划了 MCP 工作流却没确认 MCP 已加载 | 交接前验证工具可达，不要实现到一半才发现 |
| 被否的设计从零重启 | 问具体哪里不行，带收窄的约束重进，别重写 |
| 选了某地区/locale 专属的 API 变体没核对 | 写集成代码前列出所有地区/locale 差异 |
| "判断一下这个报错"走进了评估模式 | 报错语境路由到 hunt |

## Closing Pass

- 超过 8 个文件或 1 个新服务？明确承认。
- 3+ 个组件交换数据？画 ASCII 图，找环。
- 测试路径列全：happy path、错误、边界。
- 能不碰数据就回滚吗？
- 还有占位符吗？有就不算批准就绪。

## Output

**批准的方案摘要：**
- **Building**: 这是什么（一段）
- **Not building**: 明确的范围外清单
- **Approach**: 选定方案 + 理由
- **Key decisions**: 3-5 条带推理
- **Unknowns**: 只列明确推迟、写明原因和归属的项；不是模糊的空白。阻塞决策的 unknown 要在批准前回头解决。

用户批准后停。实现只在被请求时开始。输出这句收尾：

```
方案已批准。要执行：说"按计划实施"。实现后跑 /check 在合并或发布前过一遍。
```
