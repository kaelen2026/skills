# 反馈回路：构造与打磨

当 hunt 正文说"先建反馈回路再定位"时，加载这里。

反馈回路 = 一个快速、确定、agent 能自己跑出来的 pass/fail 信号。它是调试里唯一真正的工具：有了它，定位 bug 就只剩 bisection 和验证假设这些机械动作；没有它，你做的一切都是猜。这一步值得花不成比例的精力，尤其是一个人开发，没有同事帮你复现。

## 构造手段（按优先级排序）

挑第一个对当前 bug 可行的，不必从头试到尾。

1. **失败测试**：在合适的接缝（unit / integration / e2e）写一个断言坏行为的测试。最干净、最可复用，能直接变成回归守卫。
2. **HTTP 脚本**：对着 dev server 跑 `curl`，断言响应码或响应体。
3. **CLI 调用**：拿 fixture 输入跑命令，把输出 diff against 已知好的输出。
4. **无头浏览器脚本**：Playwright / Puppeteer 驱动一个最小场景。
5. **捕获 trace 回放**：把一条捕获的请求/调用 trace 隔离成单条代码路径重放。
6. **一次性 harness**：写个用完即弃的最小程序，用最少的系统搭建去触发 bug。
7. **Property / fuzz 循环**：针对"有时对有时错"的 bug，跑 1000+ 次随机输入。
8. **Bisection harness**：把 pass/fail 检查包成 `git bisect run` 能用的脚本，自动找回归点。
9. **Differential 循环**：同一输入跑在旧版 vs 新版、或配置 A vs 配置 B，对比输出。
10. **HITL bash 脚本**（最后手段）：当人工交互无法避免时，写一个把人工步骤固定下来的脚本，至少让信号可重复。

## 打磨回路本身

建出第一版后，对回路本身迭代，问三个问题：

- **能更快吗？** 目标是 2 秒的确定循环。一个时好时坏的 30 秒循环几乎没用，会让你怀疑信号而不是怀疑 bug。
- **信号能更尖吗？** 断言具体症状（确切的错误码、确切的像素、确切的字段值），而不是"它崩了"。
- **能更确定吗？** pin 住时间、seed 住 RNG、隔离文件系统、固定端口。任何一处随机都会污染信号。

## 不确定的 bug：转向提高复现率

偶发、竞态、闪烁类 bug 可能没法做到 100% 确定。这时目标从"确定复现"换成"更高的复现率"：

- 循环跑（100×、1000×）。
- 并行跑，制造资源争用。
- 加压（更高并发、更大数据、更慢磁盘）。
- 调时序（人为插入延迟、调度抖动）。

复现率从 1% 提到 50%，调试效率就不是线性提升，而是质变：你终于能在改代码后立刻知道有没有改对。
