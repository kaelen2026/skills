# TDD: 红-绿-重构，一次一颗追踪弹

一个人开发，写的代码没人 review，最容易给自己埋两类坑：测试测的是"我以为的行为"而不是真实行为，一重构就全红；以及一口气把所有测试写完再写实现，把还没理解的结构提前钉死。TDD 在这里的价值不是仪式，是给独立开发者一张随时能安全重构的网。失败模式：横切（先写全部测试再写全部实现），产出一堆测内部形状、改行为不报错、改命名却炸的废测试。

**测试验证的是通过公开接口观察到的行为，不是实现细节。** 实现可以整个换掉，测试不该跟着变。一颗追踪弹（一个测试驱动一段实现）跑通端到端，再加下一颗，每颗都吸收上一轮学到的东西。

## Outcome Contract

- Outcome: 功能以测试先行的方式建成，行为被通过公开接口的测试钉住。
- Done when: 每个被确认重要的行为都有一个测试，所有测试本回合真跑过并通过，重构后仍绿。
- Evidence: 测试命令的真实输出（红→绿各一次）、被测的公开接口、项目既有测试约定。
- Output: 通过的测试套件 + 最小实现 + 重构记录，每颗追踪弹一条红绿轨迹。

**先确认接口和要测的行为，再写第一行测试。** 你测不了所有东西，跟用户敲定哪些行为最重要（关键路径、复杂逻辑），不要给每个边界情况都铺测试。

## Core Stance

- **测行为，不测实现。** 好测试读起来像规格："用户能用合法购物车结账"。它走真实代码路径、只碰公开接口、重命名内部函数不会让它变红。一旦你 mock 内部协作者、测私有方法、或绕过接口直接查数据库验证，就是在测实现，重构必炸。
- **垂直切片，不横切。** 一个测试 → 一段实现 → 重复。绝不"先写 test1..5 再写 impl1..5"。批量写的测试测的是想象的行为，会在行为坏掉时通过、行为没事时失败。刚写完代码你才真知道哪个行为重要、怎么验。
- **GREEN 之前不重构。** 红着的时候只做一件事：写最少的代码让它变绿。重构只在全绿之后开始，且每步重构后立刻重跑测试。
- **只写当前测试需要的代码。** 不为未来的测试预埋功能，不加投机的抽象。YAGNI 在每一颗追踪弹里都成立。
- **每个测试名用项目领域词汇。** 测试名和接口词汇对齐项目的领域术语（读 README / 领域表 / 既有测试），别用 `test1`、`shouldWork` 这种空壳名。

## Pre-flight

- 先确认项目的测试命令和框架：读 `package.json` / `Cargo.toml` / `pyproject.toml` / 既有测试文件，用项目实际的 runner，别假设。
- 跟用户敲定两件事，逐项确认不要一次甩完：公开接口长什么样？哪些行为最该测？
- 区分追踪弹和后续循环：第一颗证明端到端路径打通，之后每颗加一个行为。

## Hard Rules

- **一次一个测试，红了再绿。** 看到红（且红得有道理：报的是缺失行为而不是语法错），再写实现。没看到红就写实现，等于没验证测试真的在测东西。
- **绝不横切。** 把 RED 当成"写全部测试"、GREEN 当成"写全部代码"是头号反模式。发现自己在连写第二个测试而上一个还没绿，停。
- **红着不准重构。** 重构（改名、抽函数、去重）只在全绿时做。红着重构会让你分不清是重构错了还是实现没写完。
- **不为想象的行为写测试。** 测一个你还没写实现、也没和用户确认过重要性的行为，是在凭空钉结构。先确认再测。
- **测试进项目测试套件，不是临时文件。** 一次性 scratch 脚本验证完即弃，但要长期挡回归的测试必须落进 runner 能跑到的地方。
- **不通过外部手段验证。** 别用直接查 DB、读私有字段、断言内部调用次数来"验证行为"。只通过公开接口断言可观察的结果。

## Gotchas

| 出过的问题 | 规则 |
| --- | --- |
| 一口气写完五个测试再写实现 | 垂直切片：一个测试一段实现，逐颗推进 |
| 重命名内部函数，一片测试变红 | 那些测的是实现不是行为，重写成只碰公开接口 |
| 没看到红就写了实现 | 先确认测试会红（且红在缺失行为上），再写代码 |
| 红着的时候顺手抽了个函数 | GREEN 之前不重构 |
| mock 了内部协作者，重构后测试无意义 | 走真实代码路径，只在系统真实边界处 mock |
| 给每个边界情况都写了测试，时间耗光 | 跟用户敲定关键路径，不追求穷尽 |
| 测试名是 `test_works` | 用项目领域词汇命名，读起来像规格 |

## Closing Pass（每颗追踪弹收尾自查）

- 测试描述的是行为，不是实现？
- 只用了公开接口？
- 内部重构后这个测试还会活着？
- 代码是通过这个测试所需的最小量？
- 有没有偷加投机功能？

全绿后再看一遍重构机会：去重、把复杂度移到简单接口背后（深化模块）、自然处套用 SOLID。每步重构后重跑测试。

## Output

```
behavior pinned:  [测了哪个行为, 通过哪个公开接口]
tracer bullets:   N (每颗: 测试名 -> 实现)
tests:            [命令] -> 红 X 次观察到, 绿 Y 通过
refactor:         [重构了什么, 全绿后做] 或 none
remaining:        [还没测、已与用户确认推迟的行为] 或 none
```
