# 架构盘点输出：候选清单与 HTML 呈现

何时加载：Survey 模式下候选较多、或用户想要可视化对比时。候选只有一两个时直接在终端给要点即可，不必出 HTML。

## 每个候选的字段

无论终端还是 HTML，每个深化候选都给齐：

- **Files**：涉及的源码路径（grep 查实，不凭印象）。
- **Problem**：摩擦在哪，用领域词汇 + 架构词汇描述（哪个模块浅、哪里知识分散、什么漏耦合）。
- **Solution**：建议的深化方向（把什么行为收到哪个更小的接口/seam 背后）。注意：这一步只在用户已选定该候选后才展开到接口设计；盘点阶段保持方向性描述。
- **Benefits**：深化后局部性怎么提高（改动/bug/知识怎么集中）。
- **Before / After**：当前依赖结构 vs 深化后结构的对比（图或要点）。
- **推荐强度**：`Strong`（摩擦明确、收益大、风险低）/ `值得一试`（有收益但要权衡）/ `投机`（凭直觉、需先验证）。

收尾给一个 **Top 推荐**（挑一个 + 理由），然后问用户想深入哪个，不要替用户决定动手。

## HTML 呈现（候选多或要可视化时）

写一个自包含 HTML 文件到系统临时目录：

- 路径：Unix 用 `$TMPDIR` 或 `/tmp`，Windows 用 `%TEMP%`；文件名 `architecture-review-<timestamp>.html`。
- 样式用 Tailwind CDN，图用 Mermaid CDN。
- 每个候选一张卡片，含上面六个字段 + 推荐强度徽章（Strong / 值得一试 / 投机 用不同颜色）。
- 结尾一节放 Top 推荐。

## 可视化纪律

- 图表里的依赖/调用关系一律 `grep`/`rg` 查实，不凭印象画。错的依赖图比没有图更误导。
- 图/流程优先用 Mermaid；需要精确编排的编辑式示意图用手写 CSS/SVG。
- 这份 HTML 是临时产物（在临时目录里），不是项目长期文档。沉淀下来的稳定架构结论应进 ADR / CONTEXT.md，不要把它原样留在仓库里。
