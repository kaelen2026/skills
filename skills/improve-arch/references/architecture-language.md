# 架构词汇：找深化机会时统一用的语言

何时加载：开始盘点架构、判断模块深浅、描述候选之前。这套词汇让"哪里该重构"的判断可被复述、可被检验，不退化成主观吐槽。

## 核心词

- **Module（模块）**：任何有"接口 + 实现"两面的东西。函数、类、文件、包、服务都算。不是只有大组件才叫模块。
- **Depth（深度）**：接口上的杠杆，一大堆行为藏在一个小接口背后，就是深。深模块用很小的认知成本换很多功能。
- **Shallow（浅）**：边界上暴露的复杂度几乎等于内部隐藏的复杂度。浅模块没真正帮你隐藏什么，调用方还得懂它内部，等于白加一层。
- **Seam（接缝）**：接口所在的位置，一个能在不就地改实现的前提下改变行为的地方。重构常是在挪接缝或新增接缝。
- **Locality（局部性）**：深度带来的回报，改动、bug、知识都集中在一处。局部性高 = 改一件事只碰一个地方。

## Deletion Test（删除测试）

判一个模块是真深还是该拆的关键提问：

> 删掉这个模块，复杂度是**消失**了，还是只是**摊到各个调用方**身上？

- 摊到调用方（每个调用方都得自己重新实现它做的事）→ 它在真正隐藏复杂度，值得留，可能值得深化。
- 凭空消失（删了也没人需要补回那些行为）→ 它本就没干活，是个浅包装/无用抽象,该删。

deletion test 跑出结论再下"浅/深"的判断，不要靠观感。

## 找摩擦时看的信号

- **知识分散**：理解或修改一件事，得读散落在多个文件里的片段。
- **浅模块**：接口和内部一样复杂，调用方还得懂它内部细节。
- **不可测的纯逻辑**：本可以是纯函数（入参进、值出）的决策，和可变状态/IO 纠缠在一起没法单测。
- **漏耦合**：抽象的缝里漏出实现细节，调用方依赖了不该依赖的内部。

## 用词纪律

描述候选时，把上面的架构词汇和项目自己的**领域词汇**（CONTEXT.md / README / 领域表里的真实术语）配合使用，例如"审稿队列的状态转移是个浅模块，深度该收到一个 seam 背后"。绝不用 "the service""this component""some util" 这类通用词，它们描述不出摩擦具体在哪。
