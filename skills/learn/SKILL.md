# learn: 把一堆材料变成你能用的一份东西

独立开发者要进一个陌生领域（新框架、新算法、一个想动手的方向）时，没有同事帮你筛资料、没人陪你把概念讲明白，最容易的偷懒是收 30 篇二手解读、囫囵糊成一篇看着完整其实没消化的文章。这个 skill 用一套收集→消化→提纲→填充→精炼→自审的流程，强迫你建出真正的心智模型，产出一份可信的参考/笔记/文章。它支持你的思考，不替你思考。失败模式：拿二手解读当一手来源、Phase 2 写了摘要却没建模型、把项目评审糊成通用规则。

边界：单个 URL 只需取回属于 read；需要摘要/分析的单 URL 可以用 read 当取回步骤，但最终答案要满足那个摘要/分析请求。learn 是给"多来源研究产出一份新的结构化输出"用的。

## Outcome Contract

- Outcome: 陌生材料变成一个可靠的心智模型、参考、文章或笔记，用户能直接用。
- Done when: 一手来源已收集或提供，矛盾被显式处理，最终结构教会了这个主题且不掩盖不确定性。
- Evidence: 来源 URL 或文件、取回内容、消化笔记、提纲决策、对照请求的自审。
- Output: 研究笔记 / 提纲 / 发布就绪草稿 / 权威参考，按所选模式。

## Core Stance

- 一手来源优先。提出关键思想的论文、官方实验室/产品博客、动手者的帖子、"从零搭一遍"的仓库。二手解读不是来源。
- 消化是建模型，不是写摘要。能摘要不等于消化了。
- 收得多不等于研究得好。Phase 2 砍掉大约一半。通用智慧不值得提炼。
- 矛盾要摆出来。两个来源在一个事实上冲突，记下双方立场和各自证据，不要静默选一个。
- 卡住是信号。一节难写说明那个子主题的模型还弱，回去补那个子主题，不是补整篇。

## Pre-check

检查 `read` 和 `write` 两个 skill 是否装了（找它们的 SKILL.md）。缺了告警，不阻断：

- 缺 `read`：Phase 1 取回退回原生 `WebFetch`/`curl`，付费墙、JS 重页、中文平台页的覆盖率下降。
- 缺 `write`：Phase 5 去 AI 味退回手动扫，Phase 1-4 不受影响。

## 选模式

用环境的原生提问/确认机制让用户确认模式：

| 模式 | 目标 | 入口 | 出口 |
| --- | --- | --- | --- |
| Deep Research | 把一个领域弄懂到能写 | Phase 1 | Phase 6：发布就绪草稿 |
| Quick Reference | 快速建可用心智模型，不打算写文章 | Phase 2 | Phase 2：只出笔记 |
| Write to Learn | 已有材料，用写作逼自己理解 | Phase 3 | Phase 6：发布就绪草稿 |
| Canonical Article | 一篇把主题讲透到读者不用再搜别的 | Phase 1 | Phase 6：单一权威参考 |

不确定就建议 Quick Reference。

**Activate when**（Canonical Article）: "一篇就够""一站式参考""整理成长文"，或用户要一篇权威参考。读完后没人需要再搜这个主题。在 Deep Research 之上额外要求：每个主要子主题独立成节不留脚注、给可运行的实例而非只讲原理、覆盖常见错误与规避、加 3-5 个最深来源的「延伸阅读」并标出最佳起点、Phase 6 自审须确认"读者能仅凭这篇就实现/理解吗"。

## 六阶段

1. Collect。三步有序不合并：Discover（用搜索插件或原生网搜画出图景，深搜 2-3 个最有希望的子主题，只出 URL 列表不取内容）→ Fetch（每个 URL 走 read；它管 proxy 级联、付费墙识别、平台路由；缺 read 才退回原生取回并接受覆盖下降）→ File（把 read 存的文件按子主题归档，移动不重取）。目标：博客 5-10 个来源，深度技术综述 15-20 个。
2. Digest。逐篇读全、留好的、砍不好的，本阶段结束砍掉约一半。关键断言入提纲前问三关：是否在同一来源的两个不同语境出现？这个框架能否预测来源对新问题会怎么说？是这个来源独有还是任何专家都会这么说？过两三关入提纲，过一关算背景，过零关砍掉。两来源在事实上冲突则记双方与证据。
3. Outline。逐节标注它取材自哪些来源。一节没有来源，要么它不该在，要么得先去找一个来源。提纲不稳不进 Phase 4。
4. Fill In。逐节填。一节难写就回 Phase 2 补那个子主题（不是补整篇）。卡住信号：开头句改了三遍以上仍不定、整节只靠单一来源且无法交叉验证、需要 Phase 1 没收集的新来源、写出的断言你无法对人讲清。
5. Refine。带明确 brief 过稿：删冗余不改原意与声音、标出论证不顺处、找出漏洞（概念在解释前就用了、断言缺来源）。只编辑，不摘要用户没写的节，不从零起草新节。然后去 AI 味：装了 write 就调用它，没装就手动扫填充词、二元对比、戏剧化碎句、滥用副词。
6. Self-review。用户线性通读全文（不用 AI），标出别扭处、修、再读，至少两遍。从头读到尾干净即就绪。用户确认就绪后停，不上传不发布不分发任何动作，除非明确要求。

## 把对话/评审当原材料

**Activate when**: 输入是最近的对话、项目评审、打分卡或诊断报告。
当成原材料处理，不要直接复制进文档：优先已蒸馏的摘要/记忆条目/评审输出，只为核实争议细节才开原始记录；编辑长期指南前先建候选矩阵（来源/项目、重复失败、可迁移规则、目标层、证据数、脱敏风险），只提拔有跨源支持或同项目族重复失败的候选；提取重复的工作流失败、不变量、验证面；丢掉带日期的行号、当前分数式表述、私有路径、单机配置、仓库专属命令（除非输出就是给同一仓库）；把每条耐久教训映射到目标层（项目文档、共享规则、skill references、确定性脚本）；证据片段只留作自己的笔记，不要把原始对话历史贴进最终产物。

## Gotchas

| 真实翻车 | 规则 |
| --- | --- |
| 收了 30 篇二手解读而非一手来源 | Phase 1 瞄准论文、官方博客、动手者的仓库，摘要不是来源 |
| read 装着却用原生取回/curl 取 URL | Phase 1 取回不可选，read 管 proxy 级联、付费墙识别、平台路由，绕过会静默丢覆盖 |
| 把一篇有说服力的解读当成了 ground truth | 问：它在同一来源的两个不同语境都出现吗 |
| Phase 2 写了摘要而非把概念讲透 | 消化是建模型，摘要不是消化 |
| 用户说就绪后 AI 主动提出帮忙上传发布 | 在确认处停，发布是用户的动作不是你的 |
| 把项目评审糊成通用规则没过滤 | 只提拔重复的工作流行为，项目专属命令/路径/安全约束留在那个项目 |

## Output

按所选模式返回研究笔记、提纲、发布就绪草稿或权威参考。用户确认就绪后停。
