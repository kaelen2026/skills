# 决策记录模板

正文不必加载这个文件，只有需要完整字段或示例时再读。一条记录的核心永远是三件事：选了什么、否了什么及原因、何时重审。其余字段可省。

## 文件位置

优先追加到项目已有的决策文件：`DECISIONS.md`、`docs/decisions/`、ADR 目录。都没有就在项目根建一个 `DECISIONS.md`，最新的记录追加在底部，不动旧记录。

## 最小记录（默认用这个）

```md
## [YYYY-MM-DD] [一句话标题]

- 选了什么：
- 否掉了什么 / 为什么：
- 当时的前提：
- 何时重审：
```

## 完整字段（取舍重、影响面大时用）

```md
## [YYYY-MM-DD] [一句话标题]

- 选了什么：一句话讲清最终选择。
- 否掉了什么 / 为什么：当时认真考虑过又放弃的备选，逐条写原因。只写真考虑过的，不要凑数。
- 当时的前提：这个决策依赖的约束。数据量、用户数、时间预算、商业模式、所依赖的库/服务/平台。前提是未来判断"还成不成立"的锚。
- 已知代价：明知选了它要承担什么（性能、维护、锁定某依赖）。写下来，免得未来当成意外。
- 何时重审：触发条件。例如"日活过 1000""出现第二个调用方""某依赖宣布弃用""开始收费"。没有触发条件就把这条降级成不必记。
- 相关：PRD / 方案 / commit / 代码位置的路径或链接。引用，不复制。
- 取代：若推翻了之前某条记录，写明取代哪条（日期+标题）及为什么。
```

## 示例

```md
## 2026-06-17 用 SQLite 而不是 Postgres 做本地存储

- 选了什么：单文件 SQLite，应用内嵌。
- 否掉了什么 / 为什么：Postgres——单用户桌面应用，多一个要装要管的服务，不值；云 KV——需要联网，离线不可用。
- 当时的前提：单机、单用户、数据量在十万行级、无并发写。
- 已知代价：多设备同步要另做；并发写会成为瓶颈。
- 何时重审：要做多设备同步，或出现并发写需求，或数据量进千万行。
- 相关：src/db/schema.ts，方案见 /tmp/think-storage.md
```

## 写记录时的判断

- 不确定要不要记：有过真实取舍 / 改起来有成本 / 依赖可变前提，三者沾一个就记，否则不记。
- 记录在膨胀：说明在写过程而非判断，砍回三件核心。
- 半年后回看一条记录还是看不懂当时为什么：缺的几乎总是"否掉了什么"或"当时的前提"，补这两栏。
